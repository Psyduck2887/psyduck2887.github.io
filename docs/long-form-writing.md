# 长文书写规则

适用于**稍长、结构稍复杂**的技术/观点文。短文不强制套本规则。

**优先级**

1. 以本文规则为准  
2. 其次参考范文：`source/_posts/agent-automation.md`（《Agent？自动化？》）  
3. 若范文已删除或不存在，只按本文规则写即可  

---

## 1. 文首

- 可放一张说明主旨的图（可选），放在正文最前。  
- 图用相对路径，例如：

```markdown
![简短说明](/images/your-cover.png)
```

---

## 2. 先说结论

- 开篇先给结论，让人扫一眼就知道立场。  
- **不要**用 1. 2. 3. 或一长串项目符号堆结论。  
- 用短段落写清：对谁、主张什么、为何（常与成本/场景有关）。

示例：

```markdown
## 先说结论

对公司，……更像关节。  
对个人，……更像管家。

两者都和成本有关：……
```

---

## 3. 短目录

- 结论之后、展开之前，加一段**短目录**。  
- 只链到后文带序号的大节，条目宜少（大约 3～6 条）。  
- 锚点与标题一致，保证能跳转。

示例：

```markdown
## 目录

- [1. 第一节](#1-第一节)
- [2. 第二节](#2-第二节)
- [3. 第三节](#3-第三节)
```

（生成后若锚点与预览不一致，以 `hexo generate` 后的标题 id 为准改链接。）

---

## 4. 正文章节与序号

- 展开部分的大节用序号：`## 1. …` `## 2. …`  
- 小节可用 `### 1.1` / `### 3.2`，不宜过深。  
- 需要对照时用生活化短例子即可，少堆术语。

---

## 5. 表格

- 可以用表，风格保持**素净**：白底/浅底、细边框。  
- 表头与单元格**同一字号、同一正文字体**。  
- 需要强调的词可少量使用行内 `` `code` ``（黑底强调）；**不要通篇刷**。  
- 主题已统一表样式时，正文里正常写 Markdown 表即可。

示例：

```markdown
| 做法 | 意思 |
|------|------|
| 限制工作区 | 只能动某个目录 |
| 工具白名单 | 只允许 `echo`，不允许 `rm` |
```

---

## 6. 收尾

- 最后一节可编号（如「5. 收个尾」），用几句话收回观点。  
- 可用一张小表对照，仍遵守第 5 节表格约定。

---

## 自检（长文）

- [ ] 有「先说结论」，且结论区无大序号列表  
- [ ] 结论后有短目录，链接可跳  
- [ ] 正文大节有序号  
- [ ] 表格素净、字号统一，强调很少  
- [ ] 未把短文硬套成本结构
